# 私有源码仓 + 公共 Pages 发布仓

这套方案的目标不是“绝对防拷贝”，而是把真正有价值的源码、提交历史、注释和工程结构留在私有仓，只把打包后的静态产物公开出去。

## 推荐结构

- 私有源码仓：`arrow-xiaoxiaoxiao-src`
- 公共发布仓：`arrow-xiaoxiaoxiao-site` 或 `wkylin.github.io`

建议把现在这个已经公开过源码的仓库迁成私有源码仓，再新建一个全新的公共发布仓。

原因很直接：

- 现在这个仓库已经是公开状态，历史提交里已经暴露过源码。
- 即使之后改成私有，别人之前 fork 过或拉取过的副本也不会消失。
- 全新的公共发布仓可以只保留 `dist` 产物，不带源码历史。

## 为什么要拆仓

公开 Pages 站点本身没有问题，问题在于：

- public 仓会暴露完整源码
- public 仓会暴露提交历史
- public 仓会暴露目录结构和实现思路
- public 仓更容易被直接 fork 后二次包装

拆仓后，别人最多看到打包后的前端资源，拿不到你的原始工程。

## 推荐迁移步骤

1. 新建一个私有仓，作为后续唯一开发仓。
2. 把当前项目完整推到这个私有仓。
3. 新建一个全新的公共仓，只用于放构建结果。
4. 在公共仓开启 GitHub Pages。
5. 以后所有开发、迭代、提交都只在私有仓进行。
6. 发布时由私有仓把 `dist/` 推到公共仓。

## 最稳妥的发布方式

### 方式 A：最简单，手动发布

适合先快速切换，不依赖额外 Token。

流程：

1. 在私有源码仓开发。
2. 本地执行 `pnpm build`。
3. 把 `dist/` 内容同步到公共发布仓。
4. 推送公共仓，触发 Pages 更新。

优点：

- 最直观
- 风险最低
- 不需要额外配置跨仓库权限

缺点：

- 每次发布多一步人工操作

### 方式 B：更顺手，私有仓自动发布到公共仓

适合后续稳定迭代。

流程：

1. 私有源码仓保留全部源码和当前工作流。
2. 在私有仓 GitHub Actions 里构建 `dist/`。
3. 通过专用 PAT 或 Deploy Key，把 `dist/` 推送到公共发布仓的 `gh-pages` 分支。
4. 公共仓只保留静态文件和 Pages 配置。

优点：

- 日常发布体验更顺
- 只需要推私有仓

缺点：

- 需要额外配置仓库权限和 Secrets

## 一套可直接照抄的 Actions 方案

下面这套配置适合：

- 私有源码仓负责开发
- 公共仓负责 `gh-pages`
- 你希望以后只推私有仓，就能自动更新站点

### 仓库准备

1. 新建私有源码仓，例如 `wkylin/arrow-xiaoxiaoxiao-src`
2. 新建公共发布仓，例如 `wkylin/arrow-xiaoxiaoxiao-site`
3. 在公共发布仓开启 GitHub Pages
4. Pages Source 选择 `Deploy from a branch`
5. 分支选择 `gh-pages`，目录选择 `/ (root)`

### Token 准备

1. 在 GitHub 新建一个经典 PAT 或细粒度 PAT
2. 这个 Token 至少要有对公共发布仓的写权限
3. 把 Token 存到私有源码仓的 Secret 里，名字建议叫 `PUBLIC_SITE_PAT`

### 工作流文件放哪

把示例工作流放到私有源码仓：

- [docs/examples/deploy-public-site-from-private.yml](./examples/deploy-public-site-from-private.yml)

正式启用时，把它复制成：

- `.github/workflows/deploy-public-site.yml`

然后把下面这项改成你自己的公共发布仓名：

- `external_repository: wkylin/arrow-xiaoxiaoxiao-site`

### 这套工作流会做什么

1. 拉取私有源码仓代码
2. 安装依赖
3. 执行 `pnpm build`
4. 把 `dist/` 推送到公共发布仓的 `gh-pages` 分支
5. GitHub Pages 自动更新线上站点

### 这套做法的边界

它能保护的是：

- 私有源码
- 提交历史
- 工程结构
- 注释和完整实现细节

它保护不了的是：

- 浏览器里最终下载到的打包产物
- 静态前端能被反编译的大致逻辑
- 完全在前端实现的核心算法

所以如果后面有这些内容，应该继续往后端搬：

- 排行榜
- 成绩验签
- 活动配置
- 每日挑战生成规则
- 用户数据
- 管理能力

## 当前仓库的现实情况

如果继续直接把当前 public 仓当源码仓使用，会一直暴露：

- 原始 TypeScript / React 源码
- 提交历史
- UI 和玩法演进过程
- 未来新增功能的实现细节

如果你的目标是“页面能公开访问，但源码尽量别被直接拿走”，那就不要继续把开发主仓放在 public。

## 你接下来最该做的事

1. 先把当前开发主仓迁到 private。
2. 再新建一个干净的 public 发布仓。
3. 后续只把 `dist` 发布到 public 仓。
4. 如果以后接排行榜、验分、账号、活动配置，把这些逻辑放后端，不要继续只放前端。
